# Security Policy

## Status and supported versions

Aegis is pre-release MVP software. Until the first tagged release, only the current `main` branch receives security fixes. No released version is presently supported for production use.

## Reporting a vulnerability

Do not disclose suspected vulnerabilities in a public issue, discussion, pull request, recording, or chat log. Use GitHub private vulnerability reporting from this repository's **Security** tab when it is enabled. If that facility is unavailable, stop before sending exploit details or secrets: the repository owner must publish a private security contact. That owner decision is an unresolved launch blocker; this project does not invent a contact address.

Include the affected revision, impact, prerequisites, a minimal reproduction, and whether credentials or third-party systems may have been exposed. Never include live credentials.

## Response process

A maintainer should acknowledge a private report, reproduce and classify it, coordinate a fix and regression test, prepare an advisory, and agree on disclosure timing with the reporter. Response-time commitments will be published only after maintainers establish a staffed reporting route.

## Current security boundaries

Aegis authenticates the configured local principal from OS identity, or Linux Unix-socket callers from `SO_PEERCRED`. Identity and authority are established outside the model: prompts, profile or stanza names, model conclusions, bearer tokens, and requested stanzas are not authentication. Each operational session binds to exactly one authorized trust stanza; zero matches and multiple matches deny, and authority is never unioned across stanzas. A stanza or material-authority change requires a new mandate and clean session. Bearer tokens authenticate API transport only. Exact charter and plan digests bind single-use approvals. Operational sessions use fresh Hermes processes and disposable homes, and receive only explicitly resolved environment-backed provider credentials and toolset launch arguments.

The optional local credential authority encrypts each immutable value before bbolt persistence with a fresh XChaCha20-Poly1305 DEK and wraps each DEK under a versioned KEK held outside the database. Administrative CLI operations require the configured OS principal, use no-echo or exact-stdin intake, return metadata only, enforce strict file ownership/modes and exact deployment bindings, and emit metadata-only audit events. On Linux, the optional broker adds one typed `github.get_repository.v1` operation over a pathname socket. It combines pre-body `SO_PEERCRED`, a session-bound capability, current mandate/charter/runtime identity, exact `github/read` binding and `github-api` destination, then applies the value internally and returns a sanitized field allowlist. It has no GetSecret or generic proxy endpoint. See `docs/CREDENTIAL_BROKER.md`.

These controls are not a host filesystem, network, container, or VM sandbox. bbolt does not provide encryption or RBAC, Go does not guarantee plaintext-memory zeroization, logical revocation does not erase backups/free pages, and local root can inspect usable plaintext. The broker requires distinct production service/runtime identities and is not model-visible until Hermes can verify one generated bridge without ambient MCP/plugins. Selective deployment projections remain unimplemented; Hermes still uses environment-backed provider bindings. Hermes profiles and homes isolate runtime state, not host authority. Local audit records are hash-linked and have signed checkpoints, but are not externally tamper-proof unless checkpoint retention and append authority are deployed across separately protected boundaries. Toolset launch-argument verification is not individual-tool runtime attestation. See `docs/THREAT_MODEL.md`.

The built-in manager owns terminal input, rejects bare non-terminal invocation, scans bounded source-aware content, strictly decodes one closed response/proposal envelope, and routes Hermes through an expiring, one-use-per-turn capability to an exact-model loopback proxy that scans serialized requests and complete bounded responses. Ordinary and Linux protected terminal reads are context-cancellable; protected intake restores the captured terminal state on success, failure, cancellation, and interrupted confirmation without echoing partial values. Bare startup owns Ollama readiness/load/unload, proxy and disposable safe-mode Hermes gateway lifecycle, exact certification/digest checks, rollback, metadata-only receipts, proposal confirmation, and protected no-echo mutations. All exits converge on bounded idempotent cleanup. The first SIGINT/SIGTERM requests graceful shutdown; a second signal uses the operating system's default action so cleanup cannot trap the process indefinitely. Scanner panic/timeout, malformed protocol, replayed or expired proxy calls, alternate models, public Ollama endpoints, certification drift, digest drift, and unsupported cancellation-safe protected intake fail closed. No real model certification has been performed by the project, so no artifact is a default and normal example configuration degrades honestly.

Model onboarding is a principal-authenticated, installed-only discovery and exact configuration transaction. The registry has no default; endpoints must be loopback; candidate name and immutable digest must match; previews disclose ownership and copy behavior plus original/result configuration digests; and publication requires literal confirmation, rechecks the original digest, validates the generated configuration, syncs a mode-`0600` temporary, renames atomically, and syncs the directory. It performs no model pull, registry lookup, import, copy, certification, or activation.

The self-updater trusts GitHub's release API and HTTPS delivery for this repository, rejects API and untrusted download redirects, permits only the bounded HTTPS GitHub release-asset redirect required by published assets, requires exact official-repository and stable publication metadata, and then requires the downloaded archive to match that release's `SHA256SUMS` entry before an atomic same-directory replacement. It never treats a Git tag as a published release and refuses downgrades. A checksum delivered beside an archive is corruption and consistency protection, not an independent signature or transparency proof. Package-manager installations should be updated with their package manager. Release recovery separately verifies the existing annotated tag signature, exact tag object/peeled commit, reproducible changelog-only release commit, and allowed `origin/main` relationship; it never force-pushes or mutates an existing tag.

First-run principal configuration is deterministic application code, not model authority. One typed resolver derives literal `~/.argis/aegis.yaml` and `~/.argis/state` from the authenticated owned home; XDG variables do not change local defaults. Aegis previews exact paths and bytes, requires `yes`, creates the root mode `0700`, and atomically publishes configuration mode `0600`. Existing malformed, group/other-readable, symlinked, ambiguous, or unrecognized partial artifacts fail closed. Exact legacy defaults produce `legacy-layout-detected`, and canonical plus legacy artifacts deny rather than merge. Linux `aegis migrate-layout` requires a real terminal, digest-bound plan, exact `migrate aegis to ~/.argis`, verified no-follow copy/publication, and descriptor-anchored cleanup; unsupported platforms deny automatic migration before mutation.

`aegis reset` is destructive principal administration intended for development/testing. It operates before normal service construction, prints an exact plan and irreversible credential/audit warning, requires a real terminal and literal `reset aegis`, then reauthenticates and re-inventories immediately before mutation. It rejects external-home, root, home, repository, symlinked, hard-linked, incorrectly owned/mode, unknown, and changed artifacts. Exact legacy state beneath an external mode-`0775` XDG parent is removed through no-follow descriptors bound to previewed device/inode identity; the external parent is not chmodded or called an artifact, and the empty Aegis child can be retained. Local development authority/KEK deletion additionally requires state-root containment and Aegis format/schema evidence. Systemd and external custody, normal Hermes/Ollama assets, external systems, and all downloaded models are preserved. Deletion across independent roots is not atomic: failures are incomplete and separate backups remain required.
